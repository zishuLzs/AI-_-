# 任务2 代码审查报告 v1

## 审查结论

当前实现距离“最完善的答案表现”还有明显差距，而且不只是“效果一般”，而是存在多处会导致评测直接失分甚至接近 `0 分` 的结构性问题。

最严重的问题不是算法不够强，而是 `提交契约没有对齐`、`数据库接入协议不对`、`群体统计题路径被主入口提前拦死`。这意味着即使部分公式和模板看起来正确，整份提交在真实评测环境里也很可能无法正常工作。

如果按当前版本原样提交，我的判断是：

1. 很可能无法通过 `python3 run.py "问题"` 的基础调用要求。
2. 很可能无法连接评测环境的 MySQL 数据库。
3. 很可能在示例题 Q2、Q4 这类无客户 ID 的统计问题上直接答错。
4. 多轮记忆逻辑也没有完全满足“区分假设与观点”的硬要求。

换句话说，这不是一个“已经接近提交、再调一调细节”的版本，而是一个 `需要先补齐评测接口与核心约束，再谈优化表现` 的版本。

### 4. 代码显式维护跨问题全局状态，不符合并行评测与独立作答要求

题目开发说明明确写了：

3. 如果后续接成线程内并发，也没有任何线程安全保障。

这类问题最糟糕的地方在于：`你可能本地测得通，线上却因为状态串题而随机失分`。

## 二、高优先级问题

### 5. “假设”和“观点”的区分只做了一半，仍然会污染长期记忆

题目要求：

1. `如果 / 假如 / 假设` 是临时假设，不应影响后续问题。
2. `认为 / 想要 / 预期` 是长期观点，后续应继承。

当前 [tools/router.py](/Users/lzs/Desktop/个人资料/求职/招行数字金融训练营/ai数据科学家/任务2/agent_skeleton/tools/router.py:52) 的问题在于：

1. `_extract_preferences()` 不区分问句是否处于假设语境。
2. 只要问题里出现 “每月生活费”“每月需要支出”，就会把金额写成长期 `preferences`。
3. `_extract_scenario()` 只处理通胀和额外储蓄，根本没有处理“假设退休目标月支出”这种临时目标。

举例：

`如果客户 V500002 退休后每月想要 1.5 万元生活费，她的缺口是多少？`

按题意，这里的 `1.5 万元` 应只在当前轮生效。

但按当前实现，它会被写入长期 `preferences["retirement_goal_monthly_expend"]`，后续问题会继续沿用，明显违规。

这是一个硬逻辑错误，不是文风问题。

## 三、中优先级问题

### 9. 资产配置引擎离“最小风险可行解”还有明显距离

当前 [tools/allocation_engine.py](/Users/lzs/Desktop/个人资料/求职/招行数字金融训练营/ai数据科学家/任务2/agent_skeleton/tools/allocation_engine.py:15) 的做法是：

1. 枚举 `10%` 步长权重。
2. 找能覆盖缺口的组合。
3. 在可行组合中选 `组合收益率最低` 的那个。

这不是严格意义上的“风险最低”。

因为：

1. `收益率低` 不等于 `风险最低`。
2. 没有显式风险评分函数。
3. 没有生命周期约束上限，例如临近退休是否要限制权益比例。
4. 没有处理现金储备底线。
5. 没有体现题面提到的税务规划、风险预防、长线投资等更细策略。
6. 年金险被当作普通复利资产处理，忽略其领取方式与现金流特征。

这会让建议书看上去“有组合”，但离优秀答案还差一层真正的策略解释和业务合理性。

## 四、距离最完善答案的差异总结

和一个真正“高分、稳健、适合正式评测”的答案相比，当前版本主要差在五个层面：

1. `提交协议层`
   还没有对齐评测入口、环境变量、数据库连接和并行要求。

2. `通用问题覆盖层`
   仍然是示例题导向，缺少对题目变体和自然改写的稳定支持。

3. `状态管理层`
   对话记忆虽然有了框架，但假设与观点的隔离没有做完整，且跨题持久化本身就是违规风险。

4. `策略质量层`
   资产配置和建议书仍偏“能生成”，距离“像专业养老顾问一样给出可信建议”还不够。

5. `工程可交付层`
   本地 demo 代码痕迹很重，离可提交版本还有明显收尾工作。

## 五、优先级最高的整改建议

### 第一优先级：先把“能交”修到位

1. 重写 `run.py`，补齐命令行入口。
2. 去掉本地 SQLite 依赖，改成严格按环境变量连接 MySQL。
3. 表名改为从 `TASK2_BASE_TABLE` / `TASK2_ACTION_TABLE` 读取。
4. 所有 SQL 模板去硬编码表名。
5. 去掉跨题全局状态，保证每次执行独立。

### 第二优先级：修正会直接错题的主流程逻辑

1. 取消“无 customer_id 一律拒答”的主入口拦截。
2. 单独支持群体统计题路径。
3. 修正行为表字段名与正式说明一致。
4. 给 Q2、Q4 这类题补全端到端测试。

### 第三优先级：补齐题目硬要求

1. 把“假设目标值”放入 `scenario` 而不是 `preferences`。
2. 明确每轮结束后只清临时假设，不保留跨题会话。
3. 建议书里显式汇总客户前文关注点、行为次数、缺口结果、配置理由和综合结论。

### 第四优先级：冲刺高分表现

1. 提升路由器对改写问法的鲁棒性。
2. 扩展 SQL 模板覆盖更多群体统计题。
3. 为资产配置加入显式风险评分、生命周期约束和流动性底线。
4. 为建议书加入更强的结构化证据引用。

---

## 六、最终判断

当前版本不应被视为“已经接近最佳答案，只差微调”，而应被视为：

`业务骨架初步成型，但提交协议、数据库接入、群体统计路径和状态隔离仍未达标的半成品。`

如果现在要排优先级，我的建议非常明确：

1. 先不要继续打磨文案和建议书措辞。
2. 先把 `run.py`、MySQL 接入、无 ID 统计题、状态隔离这四件事修正。
3. 这四件事不修，后面所有“更聪明的配置策略”和“更漂亮的建议书”都会被基础失分吞掉。
