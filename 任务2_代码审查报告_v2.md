# 任务2 代码审查报告 v2

## 结论

相较于 v1，当前版本已经从“明显不可提交”提升到了“具备基本提交形态”。最重要的几项修复已经完成：

1. `run.py` 已补齐命令行入口。
2. 已去掉全局 `_agent` 持久化状态，避免跨题串题。
3. 已支持从环境变量读取数据库连接参数与表名。
4. 已放开无客户 ID 的 `profile / behavior` 聚合题入口。
5. 行为表字段已从 `act_typ` 修正为 `action_typ`。
6. “假设”和“观点”在路由层做了初步分离。

这说明实现已经不再是 v1 里的“半成品”。但如果标准是“距离最完善答案还有多远”，当前版本仍有几处会显著影响得分上限，甚至在部分题型上直接答错。

最关键的剩余问题有四类：

1. `会话记忆能力被过度修复掉了`，现在每次 `run()` 都新建 agent，已经不满足题目要求的多轮历史继承。
2. `临时假设虽然被识别出来，但没有真正进入计算`，尤其是“假设退休后月支出目标”这类问题会直接算错。
3. `资产配置引擎仍然过于粗糙且引入了题目未给出的额外规则`，会拉低建议书题的表现上限。
4. `数据库默认契约和工程可验证性仍不稳`，虽然比 v1 好很多，但还没有达到最稳妥的提交质量。

## Findings

### [P1] 多轮历史记忆能力被完全关闭，违背题目第 5 条要求

当前 [run.py](/Users/lzs/Desktop/个人资料/求职/招行数字金融训练营/ai数据科学家/任务2/agent_skeleton/run.py:29) 在每次调用 `run()` 时都会执行 `_build_agent()`，重新创建：

1. `MemoryManager`
2. `IntentRouter`
3. 所有 Skill

并且 [run.py](/Users/lzs/Desktop/个人资料/求职/招行数字金融训练营/ai数据科学家/任务2/agent_skeleton/run.py:118) 明确写成“Each call creates a fresh agent”。

这修掉了 v1 的跨题串题问题，但同时把题目原始要求中的这一条也一并删掉了：

`具有保存历史会话信息的能力，能够根据前序问题中给到信息回答后续问题`

题目原文见 [任务2_养老规划Agent建设.md](/Users/lzs/Desktop/个人资料/求职/招行数字金融训练营/ai数据科学家/任务2/任务2_养老规划Agent建设.md:113)。

后果是：

1. 如果评测程序在同一解释器内连续调用 `run("问题1")`、`run("问题2")`，当前实现不会记住问题 1 中的客户偏好与目标。
2. 建议书题无法真正利用“前序用户提问中的关注要点”，因为这些状态根本不会跨调用保留。
3. 你现在满足了“并行独立”，但牺牲了“多轮继承”，这是典型的过度修复。

这是当前版本最严重的功能性缺口。

### [P1] 路由层已经把“假设退休目标月支出”写入 `scenario`，但公式层根本没用这个值

在 [tools/router.py](/Users/lzs/Desktop/个人资料/求职/招行数字金融训练营/ai数据科学家/任务2/agent_skeleton/tools/router.py:79) 中，假设语境下会把如下值写入临时场景：

1. `scenario["retirement_goal"]`
2. `scenario["retirement_goal_monthly_expend"]`
3. `scenario["inflation_annual"]`
4. `scenario["extra_monthly_saving"]`

但在 [tools/formula_engine.py](/Users/lzs/Desktop/个人资料/求职/招行数字金融训练营/ai数据科学家/任务2/agent_skeleton/tools/formula_engine.py:39) 中，退休支出计算只读取：

1. `preferences["retirement_goal_monthly_expend"]`
2. 否则按当前月支出推导

完全没有读取 `scenario["retirement_goal_monthly_expend"]`。

这会导致一种非常具体且非常危险的错题：

`如果客户 V500002 退休后每月想要 1.5 万元生活费，她的缺口是多少？`

当前实现会：

1. 正确识别这是临时假设。
2. 正确把 `15000` 放进 `scenario`。
3. 但公式引擎忽略这个假设值，继续按默认支出逻辑计算。

也就是说，`识别对了，计算错了`。

这类问题在评测里极容易被打穿，因为它正好对应题目强调的“区分假设与观点”。

### [P1] 资产配置权重离散粒度过粗，无法逼近题目示例中的高质量方案

当前 [tools/allocation_engine.py](/Users/lzs/Desktop/个人资料/求职/招行数字金融训练营/ai数据科学家/任务2/agent_skeleton/tools/allocation_engine.py:148) 使用：

1. `slots = 10`
2. 即 `10%` 为步长枚举权重组合

这意味着组合只可能是：

1. `0% / 10% / 20% / ... / 100%`

而题目示例的高质量方案是：

1. `73% 固收+`
2. `10% 现金理财`
3. `17% 年金险`

见 [任务2_养老规划Agent建设.md](/Users/lzs/Desktop/个人资料/求职/招行数字金融训练营/ai数据科学家/任务2/任务2_养老规划Agent建设.md:255)。

当前配置引擎无论如何都不可能生成这种精细比例，只能生成非常粗的近似方案。结果是：

1. 可能找不到“最小风险且恰好覆盖缺口”的真正最优点。
2. 组合往往要么偏保守、要么偏过量。
3. 建议书里的数字表现会明显不如精细求解方案。

如果目标是“最完善答案表现”，`10%` 粒度明显不够，至少应该下沉到 `1%` 或使用更直接的约束优化方法。

### [P1] 当前实现引入了题目未给出的额外业务规则，可能导致与标准答案偏离

在 [tools/allocation_engine.py](/Users/lzs/Desktop/个人资料/求职/招行数字金融训练营/ai数据科学家/任务2/agent_skeleton/tools/allocation_engine.py:22) 中，新增了：

1. 生命周期权益上限 `_EQUITY_CAPS`
2. 现金底仓 `_CASH_FLOOR = 0.05`
3. 液体资产集合 `_LIQUID_PRODUCTS`

这些规则从业务上看有道理，但题目并没有给出这类硬约束。

风险在于：

1. 你会主动排除一些本来可行、甚至更接近标准答案的组合。
2. 当评测的参考口径更偏“覆盖缺口的最低风险方案”时，你可能被自己加的规则卡住。
3. 建议书中的方案不再只是“依托产品库”，而是“依托你自定义的一套额外投顾政策”。

这类规则不是不能加，但如果要加，必须证明它不会影响示例题口径和高分答案方向。当前代码没有做到这一点。

2. `TASK2_DB_USER=test_user`

4. `TASK2_DB_NAME=cmb_contest`
5. `TASK2_BASE_TABLE=train_base_table`
6. `TASK2_ACTION_TABLE=train_action_table`

见 [任务2/README.md](/Users/lzs/Desktop/个人资料/求职/招行数字金融训练营/ai数据科学家/任务2/README.md:72)。

这不是线上必挂的问题，但它说明：

1. 代码默认运行契约和题面文档仍不一致。
2. 自测体验会和题目提供的标准环境脱节。
3. 一旦评测环境只注入部分变量，当前默认值就可能落到错误分支。

### [P2] 行为统计能力比 v1 好了，但仍然是“示例题驱动”的脆弱实现

## 优先级建议

### 第一优先级

1. 设计“可控的多轮记忆机制”，既满足题目会话继承，又不造成跨题污染。
2. 让 `scenario["retirement_goal_monthly_expend"]` 真正进入公式计算。

### 第二优先级

1. 把资产配置粒度从 `10%` 降到 `1%`，或改成优化求解。
2. 重新评估自定义的 `现金底仓 / 权益上限 / 风险评分` 是否会偏离题目标准口径。

### 第三优先级

1. 对群体行为问题做更强的语义泛化，而不是继续堆关键词。
2. 增加端到端评测样例，覆盖“历史继承 + 假设隔离 + 建议书生成”全链路。

## 最终判断

v2 视角下，这份实现已经从“不可交”进入了“可交但上限受限”的阶段。

如果现在提交，它大概率不会像 v1 那样在接口层直接失守；但如果目标是更高分，当前最大的瓶颈已经不是“入口格式”了，而是：

1. `多轮记忆能力缺失`
2. `临时假设没有真正参与计算`
3. `配置引擎过粗且额外规则过多`
4. `开放式问题的泛化能力仍弱`

这四点不解决，当前版本可以算“合格实现”，但还达不到“最完善答案表现”。
